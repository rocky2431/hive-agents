# Hive 最快路径：飞书驻场数字员工产品化方案

> 基于当前代码状态、已有架构提案、以及 `.ultra/docs/` 四份专家分析整理
> 日期：2026-03-20
> 目标：从“功能很多的 Agent 平台”收敛到“最快可成交的 agent-native SaaS”

---

## 1. 核心结论

如果只用一个原则来指导接下来 1-2 个季度的决策，那就是：

> **优先把“飞书驻场数字员工”做成一个企业愿意购买、愿意长期使用的产品闭环。**

这意味着：

- 不先追求最全的平台能力
- 不先追求最复杂的本地执行能力
- 不先追求覆盖所有行业场景
- 不先追求一次性完成全部四层权限重构

而是优先做成一个高密度闭环：

1. 企业可以安全接入
2. Agent 可以在飞书里长期驻场
3. Agent 可以自主执行有限但清晰的工作
4. 关键动作可审批、可审计、可解释
5. 管理者能理解它在做什么、以谁的身份做、为什么能做

---

## 2. 北极星定义

Hive 当前最应该被定义为：

> **一个能在企业沟通渠道中持续工作、可被管理、可被审计、可被授权的数字员工系统。**

不是：

- 通用 Bot 平台
- 低代码工作流编辑器
- 本地代码 Agent
- 文档中心

这一定义非常重要，因为它会直接决定优先级。

### 2.1 北极星产品形态

理想的一线体验应当是：

1. 管理者创建一个 Agent
2. 为它定义角色、职责、工作边界
3. 将它接入飞书
4. 设定它的自主边界和审批规则
5. 把它放进一个群、一个项目、一个工作流程里
6. 它开始持续工作、主动提醒、回答问题、汇报进展、触发任务

这个体验一旦成立，Hive 就从“平台能力集合”变成“可采购产品”。

---

## 3. 为什么这是最快路径

### 3.1 因为当前代码资产已经支持这条路

根据当前实现，Hive 已经具备这条路径最关键的骨架：

- Agent 持久身份：`backend/app/models/agent.py`
- Agent 工作区：`backend/app/services/agent_manager.py`
- 飞书渠道集成：`backend/app/api/feishu.py`
- 主动执行：`backend/app/services/trigger_daemon.py`
- 审批能力：`backend/app/services/autonomy_service.py`
- 审计基础：`backend/app/models/security_audit.py`、`backend/app/core/policy.py`
- 统一 runtime：`backend/app/runtime/invoker.py`
- 记忆系统：`backend/app/services/memory_service.py`

也就是说，最快路径不是重新发明产品，而是**把已有能力压缩成一个强闭环场景**。

### 3.2 因为这条路的购买阻力最明确

当前企业采购是否愿意推进，主要不是看你有多少工具，而是看下面几件事：

1. 能不能接企业身份体系
2. 能不能审计
3. 能不能解释 Agent 的权限边界
4. 能不能在现有工作渠道里工作

飞书驻场数字员工正好对应这四点。

### 3.3 因为这条路和你的真实使用场景一致

如果你自己的核心场景是 VC 投后、持续跟踪、群内协作、主动提醒、周期总结，那么：

- 飞书是天然工作入口
- Agent 的持久身份和主动性有真实价值
- 多 Agent 协作也有清晰落点

这比泛化做“适用于所有行业的通用 Agent 平台”更容易穿透。

## 4. 产品原则

接下来所有决策，建议都服从下面四条原则。

### 原则一：场景优先于平台完备度

如果一个能力不能显著增强“飞书驻场数字员工”闭环，就不应优先。

### 原则二：治理优先于炫技

企业愿意付费，不是因为功能最多，而是因为：

- 能接入组织
- 能解释权限
- 能审计风险
- 能持续稳定运行

### 原则三：Agent 是独立能力主体，不是员工账号影子

Hive 的 Agent 不应默认继承员工全部权限。

应当是：

- 员工与 Agent 建立关系
- 企业授予它一部分能力
- 它在受控身份下执行

### 原则四：本地执行是增强项，不是当前成交前提

Local Connector 值得设计，但不应在 PMF 前吃掉主资源。

## 5. 当前最该做的产品闭环

建议把 v1 的闭环收缩为：

## 飞书驻场数字员工

### 5.1 目标用户

- VC / 投后管理团队
- 研究团队
- 客户成功 / 运营团队
- 中小企业内部协作团队

### 5.2 核心工作流

1. 管理员创建 Agent
2. 配置职责、角色、工作风格
3. 接入飞书 Bot / 渠道
4. 配置自主边界
5. 把 Agent 放入群聊或工作流
6. Agent 持续参与：
   - 回答问题
   - 主动提醒
   - 周期性汇报
   - 文档读取与摘要
   - 跟踪待办与触发器

### 5.3 最小业务承诺

第一阶段不要承诺“万能数字员工”，只承诺：

- 长期驻场
- 有记忆
- 能主动
- 有边界
- 可追溯

## 6. Phase 1 只做什么

建议把 Phase 1 定义成一个非常克制的范围。

### 6.1 必做项

#### A. OIDC SSO

原因：

- 企业采购的入场券
- 没有它，IT 往往不会继续

建议范围：

- 先做 OIDC
- 不把 SAML / SCIM 放入 Phase 1

#### B. 审计日志产品化

现在后端已经有基础审计模型和事件写入，但还不是企业级可交付体验。

Phase 1 目标：

- 审计日志可按 Agent、用户、时间、动作筛选
- 能看关键事件链路
- 能导出
- 能清楚区分：
  - 谁发起
  - 谁批准
  - Agent 执行了什么
  - 最终结果如何

#### C. Execution Identity 最小实现

不要一上来做全系统 execution identity。

Phase 1 只建议在飞书链路上实现最小版本：

- `agent_bot`
- `delegated_user`

先解决最关键的企业信任问题：

> 这条消息到底是 Agent 自己发的，还是代表某个员工发的？

#### D. 高风险 Capability Policy

不要一上来覆盖全部工具。

先只覆盖高风险动作：

- 发消息
- 建日程
- 外部系统写操作
- 修改 soul / memory / trigger
- 删除类操作

#### E. 飞书驻场体验打磨

不是继续堆工具，而是把体验闭环打磨完整：

- Agent onboarding
- 渠道接入
- 自主边界设置
- 群内值班体验
- 周期触发与汇报体验

### 6.2 暂不做项

#### A. Full Local Runtime

明确不做。

#### B. 重型 Local Connector 实现

协议可以先设计，但实现推后。

#### C. 代码 Agent 场景

现在不进入主路线。

#### D. 自建文档编辑器

不做。

文档只作为 workspace resource 或外部文档连接器资源接入。

#### E. 四层权限全量重构

四层模型是正确方向，但 Phase 1 不应追求一次性完整迁移。

---

## 7. Phase 1 的真正目标不是“权限系统完成”，而是“企业敢用”

这点必须非常明确。

Phase 1 成功，不等于：

- 所有权限模型都设计完
- 所有工具都接入 capability system
- 所有渠道都支持 delegated identity

Phase 1 成功，应该等于：

1. 企业员工能通过 SSO 进入系统
2. 管理员能创建并投放飞书驻场 Agent
3. 高风险动作有清晰审批边界
4. 审计日志足够支撑内部解释与导出
5. Agent 能稳定地在飞书群中长期工作

如果这五件事成立，产品就具备初步销售条件。

---

## 8. 建议的 6-8 周交付顺序

### Week 1-2：企业接入门槛

交付：

- OIDC SSO 最小可用
- 企业身份接入流程
- 基本会话与租户映射

目标：

- 企业用户能无摩擦进入 Hive

### Week 2-4：审计与解释能力

交付：

- 审计 UI 升级
- 过滤、检索、导出
- 关键动作事件模型整理

目标：

- 管理员能回答“Agent 做了什么”

### Week 3-5：飞书执行身份最小实现

交付：

- 飞书 `agent_bot` / `delegated_user` 区分
- 高风险动作审计记录展示执行身份
- 最小审批联动

目标：

- 管理员能回答“Agent 是以谁的身份做的”

### Week 4-6：高风险 Capability Policy 接入

交付：

- 对关键工具增加 capability gate
- 与现有 autonomy 审批兼容
- 不破坏当前运行时

目标：

- 管理员能回答“为什么它能做这件事”

### Week 5-8：飞书驻场体验打磨

交付：

- onboarding
- 配置边界
- 群聊驻场体验
- 定期汇报和触发器样板

目标：

- 业务方愿意试用并持续使用

## 9. 我建议立即砍掉或延后的事情

为了保证“最快路径”，下面这些事建议主动延后。

### 9.1 Local Connector 实现

可以保留协议设计，不要进入主开发资源。

原因：

- 当前不是成交前提
- 安全和能力边界复杂
- 容易把产品重心拉偏

### 9.2 通用文档中心

不做独立产品线。

建议只做：

- 文档读取
- 摘要
- 搜索
- 外部文档接入

### 9.3 代码 Agent 产品化

直接延后。

原因：

- 不是当前最短 PMF 路径
- 竞品极强
- 会显著分散资源

### 9.4 全面通用化平台抽象

现在不要为了“未来所有场景”而提前引入大规模抽象。

只为当前飞书驻场数字员工闭环做足够的抽象即可。

---

## 10. 推荐的产品包装方式

第一阶段不要把自己包装成“万能 Agent 平台”。

更建议这样讲：

> Hive 帮企业雇佣和管理数字员工。
> 它们可以驻场在飞书中，持续协作、主动工作，并且所有关键动作都有边界和审计。

这个表述比：

- 多 Agent 平台
- 企业 Bot 平台
- AI 自动化平台

更有识别度，也更贴近当前代码能力。

---

## 11. 建议的 Phase 1 成功标准

建议至少用下面这些结果判断 Phase 1 是否成功。

### 11.1 产品层

- 一个新租户能在 30 分钟内完成接入和首个 Agent 上线
- 一个 Agent 能稳定驻场在飞书群中连续工作
- 管理者能清楚看懂 Agent 的权限边界

### 11.2 工程层

- OIDC 登录稳定
- 审计日志事件链路可追溯
- 高风险工具执行前有明确 gate
- 飞书 execution identity 能区分并落审计

### 11.3 商业层

- 至少拿到一批真实试用团队
- 至少出现一批持续使用的驻场 Agent
- 至少形成一个可复用销售故事

---

## 12. 对四层权限模型的现实态度

四层模型：

- Binding
- Access
- Capability
- Execution Identity

方向是正确的，我同意继续推进。

但对最快路径来说，正确策略应当是：

### 12.1 先落最小能力，不先追求全量完备

建议顺序：

1. `Execution Identity` 先落飞书
2. `Capability` 先落高风险动作
3. `Access` 保持现有 use/manage 兼容运行
4. `Binding` 先不做全面产品化

### 12.2 先让产品可卖，再让模型完美

如果一开始就做完整 schema、完整迁移、完整 UI、完整 runtime 接入，极容易变成长周期治理工程。

这不符合“最快路径”的原则。

---

## 13. Local Connector 的正确姿态

Local Connector 不是不重要，而是：

> **它是第二阶段增强项，不是第一阶段成交条件。**

建议当前策略：

- 现在定义协议方向
- 现在统一未来 workspace URI 思路
- 现在不做主实现

只有当出现明确、重复的本地执行需求时，再投入工程资源。

---

## 14. 最终建议

如果现在只能做一件最重要的事情，那不是继续扩工具，也不是继续扩场景，而是：

> **把飞书驻场数字员工做成一个真正可采购、可交付、可解释的企业产品。**

这条路的具体落点应该是：

1. 先补 OIDC
2. 先补审计产品化
3. 先补飞书 execution identity 最小实现
4. 先补高风险 capability gate
5. 最后把飞书驻场体验打磨成完整方案

只要这条线跑通，Hive 就不再只是“有很多潜力的 Agent 平台”，而会变成“企业可以真实部署的数字员工系统”。
